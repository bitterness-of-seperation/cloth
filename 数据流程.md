# 虚拟试穿 App — 数据流程文档

## 一、图片合成试穿流程

### 整体架构

```
用户浏览器 (Next.js)
    │
    ├─ 1. 上传图片 ──→ Supabase Storage
    │
    ├─ 2. 发送合成请求 ──→ FastAPI 后端 (:8000)
    │                          │
    │                          ├─ 3a. 下载图片 (从 Supabase public URL)
    │                          │
    │                          ├─ 3b. 策略1: IDM-VTON HF Space
    │                          │      (失败则降级)
    │                          ├─ 3c. 策略2: RapidAPI VTON-D
    │                          │      (未配置则跳过)
    │                          ├─ 3d. 策略3: PIL 简单贴图 (fallback)
    │                          │
    │                          └─ 4. 返回 base64 图片 + 方法名
    │
    ├─ 5. 解码 base64 → Blob
    ├─ 6. 上传结果 ──→ Supabase Storage
    ├─ 7. 写入记录 ──→ Supabase Database (tryon_results 表)
    └─ 8. 显示结果图片
```

### 详细步骤

#### 前端 (`frontend/src/app/tryon/page.tsx`)

| 步骤 | 操作 | 数据 |
|------|------|------|
| 1 | 用户选择人物照片和服装图片 | `File` 对象 |
| 2 | 并行上传到 Supabase Storage | `person-photos` 和 `clothing` bucket |
| 3 | 获取公开 URL | `https://xxx.supabase.co/storage/v1/object/public/...` |
| 4 | POST 请求到后端 | `{ person_image_url, clothing_image_url }` |
| 5 | 等待后端响应 | 可能需要 30s-120s |
| 6 | 收到 base64 结果 | `{ result_image_base64, method, elapsed_sec }` |
| 7 | 解码并上传到 `tryon-results` bucket | JPEG Blob |
| 8 | 插入数据库记录 | `tryon_results` 表 |

#### 后端 (`backend/app/services/tryon_service.py`)

| 步骤 | 操作 | 日志关键字 |
|------|------|-----------|
| 1 | 接收请求，分配 req_id | `[req_id] --> POST /api/tryon/generate` |
| 2 | 下载人物和服装图片 | `下载图片 url=... 大小=...KB 耗时=...s` |
| 3 | 验证图片格式和尺寸 | `图片尺寸 person=... clothing=...` |
| 4 | 尝试 IDM-VTON | `[策略1/3] 尝试 IDM-VTON...` |
| 4a | 连接 HF Space | `[IDM-VTON] 连接 HF Space: yisol/IDM-VTON` |
| 4b | 上传图片并推理 | `[IDM-VTON] 推理完成 耗时=...s` |
| 4c | 失败时诊断原因 | `[策略1/3] 原因: 网络无法连接/排队超时/...` |
| 5 | 尝试 VTON-D (如配置) | `[策略2/3] 尝试 RapidAPI VTON-D...` |
| 6 | 降级到 PIL fallback | `[fallback] ⚠ 使用 PIL 简单贴图模式` |
| 7 | 返回结果 | `试穿完成 ✓ 方法=... 总耗时=...s` |

### 合成策略降级链

```
IDM-VTON (HF Space, 免费, AI 合成, 效果最好)
    │ 失败原因: 网络不通 / Space 离线 / GPU 配额用尽 / 超时
    ▼
VTON-D (RapidAPI, 需 RAPIDAPI_KEY, 商业 API, 效果好)
    │ 失败原因: 未配置 key / API 返回错误 / 超时
    ▼
PIL Fallback (本地, 永远可用, 简单贴图, 效果差)
```

### 环境变量

| 变量 | 位置 | 说明 |
|------|------|------|
| `HF_TOKEN` | `backend/.env` | HuggingFace token，加速 HF Space 连接 |
| `RAPIDAPI_KEY` | `backend/.env` | RapidAPI key，启用 VTON-D 商业 API |
| `NEXT_PUBLIC_API_URL` | `frontend/.env.local` | 后端地址，默认 `http://localhost:8000` |
| `NEXT_PUBLIC_SUPABASE_URL` | `frontend/.env.local` | Supabase 项目 URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `frontend/.env.local` | Supabase 匿名 key |

## 二、AR 实时试穿流程

### 整体架构

```
用户浏览器 (全部在前端完成，无后端调用)
    │
    ├─ 1. 用户上传服装图片
    │      └─ 自动去白色背景 + 边缘羽化
    │
    ├─ 2. 开启摄像头 (getUserMedia)
    │
    ├─ 3. 加载 MediaPipe PoseLandmarker (Heavy 模型, GPU 加速)
    │
    └─ 4. 每帧循环 (requestAnimationFrame):
           │
           ├─ 4a. 绘制镜像摄像头画面到 Canvas
           │
           ├─ 4b. MediaPipe 检测 33 个身体关键点
           │
           ├─ 4c. EMA 平滑关键点 (消除抖动)
           │
           ├─ 4d. 计算身体区域四角点:
           │      ├─ 肩部角度 (atan2)
           │      ├─ 袖子延伸
           │      └─ 腰部收窄
           │
           └─ 4e. 8x10 网格细分 + 双线性插值变形绘制
```

### 关键技术点

#### 1. 服装图片预处理

```
原始图片 → 去白色背景 (阈值 230) → 边缘渐变羽化 (12px) → 处理后的 Canvas
```

- 白色背景像素 (R,G,B > 230) 的 alpha 设为 0
- 接近白色的像素 (200-230) 做渐变透明
- 四周边缘 12px 做 alpha 渐变，避免硬边

#### 2. 身体关键点

使用 MediaPipe PoseLandmarker 的关键点索引:

| 索引 | 名称 | 用途 |
|------|------|------|
| 11 | 左肩 | 衣服上边界左端 |
| 12 | 右肩 | 衣服上边界右端 |
| 13 | 左肘 | 袖子方向参考 |
| 14 | 右肘 | 袖子方向参考 |
| 23 | 左髋 | 衣服下边界左端 |
| 24 | 右髋 | 衣服下边界右端 |

#### 3. 网格变形

- 将衣服区域划分为 8x10 = 80 个网格单元
- 每个单元分为 2 个三角形，共 160 个三角形
- 每个三角形用仿射变换 (`ctx.transform`) 绘制对应的纹理区域
- 双线性插值确保网格顶点平滑过渡

#### 4. EMA 平滑

```
smoothed = prev + (current - prev) × alpha
alpha = 0.35 (越小越平滑，但延迟越大)
```

### 数据流（拍照保存时）

```
Canvas 截图 → toBlob(JPEG) → Supabase Storage (tryon-results)
                                    │
                                    └─→ Supabase Database (tryon_results 表)
```

## 三、数据库结构

### Supabase Storage Buckets

| Bucket | 用途 | 访问 |
|--------|------|------|
| `person-photos` | 用户上传的人物照片 | 公开读 |
| `clothing` | 用户上传的服装图片 | 公开读 |
| `tryon-results` | 合成结果 / AR 截图 | 公开读 |

### Supabase Database Tables

#### `tryon_results`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | uuid | 主键 |
| `user_id` | uuid | 用户 ID (关联 auth.users) |
| `person_image_url` | text | 人物照片 URL (AR 模式为空) |
| `result_image_url` | text | 结果图片 URL |
| `created_at` | timestamptz | 创建时间 |

## 四、后端日志排查指南

### 日志格式

```
时:分:秒 | 级别    | 模块名 | 消息内容
14:16:11 | INFO    | app    | [abc12345] --> POST /api/tryon/generate  client=127.0.0.1
```

### 关键日志检查点

1. **请求是否到达后端**
   - 看 `[req_id] --> POST /api/tryon/generate`
   - 没有 → 检查前端 `NEXT_PUBLIC_API_URL` 和 CORS 配置

2. **图片是否下载成功**
   - 看 `下载图片 url=... 大小=...KB`
   - 失败 → Supabase Storage bucket 是否公开

3. **使用了哪个合成策略**
   - 看 `[策略1/3]` / `[策略2/3]` / `[策略3/3]`
   - 看 `试穿完成 ✓ 方法=...`

4. **策略失败原因**
   - 看 `[策略X/3] 原因: ...`
   - 常见: 网络不通、HF Space 离线、RAPIDAPI_KEY 未配置

5. **总耗时**
   - 看 `总耗时=...s`
   - IDM-VTON 通常 30-120s，VTON-D 5-15s，fallback < 3s
